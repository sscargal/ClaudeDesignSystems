The Compute Express Link standard is quietly reshaping how we think about memory hierarchies in data centers, enabling new classes of disaggregated memory at DRAM-like latency.

Body / .prose · 17px · leading-article (1.75)

Persistent memory occupies a unique position in the storage hierarchy — faster than NVMe SSDs by orders of magnitude, yet byte-addressable like DRAM. Programming it correctly requires understanding both the hardware characteristics and the PMDK library abstractions that shield you from the low-level details of ensuring write ordering and crash consistency.

C Code — PMDK Example

c 
/* Open or create a persistent memory pool */
PMEMobjpool *pop;
const char *path = "/mnt/pmem/mypool";

if (access(path, F_OK) != 0) {
    pop = pmemobj_create(path, "MY_LAYOUT",
                          PMEMOBJ_MIN_POOL, 0666);
    if (pop == NULL)
        err(1, "pmemobj_create");
} else {
    pop = pmemobj_open(path, "MY_LAYOUT");
    if (pop == NULL)
        err(1, "pmemobj_open");
}

/* Begin a transaction to guarantee crash-consistency */
TX_BEGIN(pop) {
    TOID(struct node) n = TX_NEW(struct node);
    D_RW(n)->value = 42;
    pmemobj_persist(pop, &D_RW(n)->value,
                    sizeof(D_RW(n)->value));
} TX_ONABORT {
    fprintf(stderr, "Transaction aborted!\n");
} TX_END

Inline Code

To flush a range to persistence, call pmem_persist(addr, len). If the device is not a real pmem device, this falls back to pmem_msync(). The PMDK macro TOID_ASSIGN(oid, ptr) provides type-safe object identifiers.
